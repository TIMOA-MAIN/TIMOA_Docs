# 0013. 파일 저장 Base64 + MS SQL → AWS S3 전환

## 상태
승인됨

## 맥락
초기 채팅 파일 업로드·다운로드 기능은 파일을 Base64로 인코딩하여 MS SQL DB에 직접 저장하는 방식으로 구현되었다. 이는 빠른 구현을 위한 임시 방편이었으나 다음과 같은 문제가 명확했다.

- DB 용량 급증: 이미지·파일이 Base64로 DB에 저장되면 원본 대비 약 33% 용량 증가
- 조회 성능 저하: 대용량 Base64 데이터가 포함된 레코드 조회 시 성능 저하
- DB 백업 부담: 비정형 데이터가 포함된 DB 백업은 용량이 크고 시간이 오래 걸림
- 관심사 미분리: 정형 데이터(메시지, 사용자 정보)와 비정형 데이터(파일)가 같은 저장소에 혼재

## 고려한 옵션

### 옵션 A — Base64 + MS SQL 유지
- 장점: 별도 스토리지 설정 불필요, 구현 단순
- 단점: DB 용량 낭비, 성능 저하, 백업 부담, 안티패턴

### 옵션 B — 온프레미스 파일 서버
- 장점: 별도 클라우드 비용 없음
- 단점: 서버 용량 제한, 가용성 보장 어려움, 직접 관리 필요

### 옵션 C — AWS S3 (선택)
- 장점: 무제한 스토리지, 고가용성, CDN(CloudFront) 연동 용이, 정형/비정형 데이터 분리, 객체 단위 접근 제어
- 단점: AWS 비용 발생, 외부 서비스 의존성

## 결정
채팅 파일 업로드·다운로드를 AWS S3 버킷으로 전환한다. DB에는 S3 객체 URL만 저장한다.

## 근거
Base64로 DB에 파일을 저장하는 것은 명백한 안티패턴이다. RDBMS는 정형 데이터 관리에 최적화되어 있고, 대용량 비정형 데이터는 오브젝트 스토리지가 적합하다. AWS S3는 사실상 업계 표준 오브젝트 스토리지로 높은 가용성과 내구성을 보장하며, 추후 CloudFront CDN 연동도 용이하다.

## 트레이드오프
- 얻은 것: DB 용량 최적화, 조회 성능 향상, 정형/비정형 데이터 분리, S3 내구성·가용성
- 감수한 것: AWS S3 비용, 외부 서비스 의존성 추가

## 결과
- 채팅 파일·이미지 AWS S3 저장 완료
- DB에는 S3 URL만 저장하여 용량 최적화
