# 0014. 비정형 데이터 AWS S3 버킷 분리 결정

## 상태
승인됨

## 맥락
파일 저장을 S3로 전환([0013](./0013-file-s3-migration.md))하면서, 비정형 데이터 전반을 어떻게 관리할지 전략을 수립해야 했다.

TIMOA에서 비정형 데이터의 종류는 다음과 같다.
- 채팅 첨부 파일 (이미지, 문서, 압축 파일 등)
- 사용자 프로필 이미지
- 문서 채널 내 이미지 블록

이 모든 비정형 데이터를 하나의 S3 버킷에서 관리할지, 용도별로 분리할지 결정이 필요했다.

## 고려한 옵션

### 옵션 A — 단일 S3 버킷 통합 관리 (선택)
- 장점: 관리 단순, 비용 효율적, 접근 제어 정책 일원화
- 단점: 버킷 내 객체 수 증가 시 관리 복잡도 증가 (폴더 구조로 보완 가능)

### 옵션 B — 용도별 S3 버킷 분리
- 장점: 용도별 독립적 접근 제어, 버킷별 정책 세분화
- 단점: 버킷 관리 복잡도 증가, 비용 증가, 예산 제약에 부적합

## 결정
단일 S3 버킷을 사용하되, 폴더(prefix) 구조로 데이터 유형을 분리하여 관리한다.

```
timoa-bucket/
├── chat/           ← 채팅 첨부 파일
│   ├── images/
│   └── files/
├── profiles/       ← 사용자 프로필 이미지
└── documents/      ← 문서 채널 이미지
```

## 근거
예산 제약으로 다수의 버킷을 운영하는 것보다 단일 버킷 + prefix 구조가 현실적이다. 폴더 구조로 데이터 유형을 분리하면 관리가 충분히 가능하고, IAM 정책으로 prefix 단위 접근 제어도 가능하다.

S3를 비정형 데이터 전용 저장소로 분리함으로써 MS SQL은 순수하게 정형 데이터만 관리하는 관심사 분리가 완성된다. 추후 CloudFront CDN을 S3 앞단에 붙이면 파일 전송 성능도 개선할 수 있다.

## 트레이드오프
- 얻은 것: 정형/비정형 데이터 완전 분리, S3 내구성·가용성, CDN 연동 준비
- 감수한 것: 단일 버킷으로 용도별 접근 제어가 버킷 수준에서는 불가 (prefix 정책으로 보완)

## 개선 방향
인프라 환경 개선 시 CloudFront를 S3 앞단에 연결하여 파일 전송 성능을 개선한다.

## 결과
- 모든 비정형 데이터가 AWS S3로 완전 이전
- MS SQL은 정형 데이터(사용자, 메시지 메타, 프로젝트 정보)만 관리
- 관련 결정: [0003](../architecture/0003-shared-db.md), [0013](./0013-file-s3-migration.md)
