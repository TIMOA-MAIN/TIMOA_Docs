# 0010. 음성 통화에 P2P 대신 SFU(MediaSoup) 채택

## 상태
승인됨

## 맥락
음성 통화 기능 구현 시 WebRTC의 동작 방식을 선택해야 했다. WebRTC는 P2P, MCU, SFU 세 가지 방식으로 구현할 수 있으며 각각 특성이 다르다.

TIMOA의 음성 통화 요구사항은 다음과 같았다.
- 다중 사용자(팀 단위) 동시 음성 통화
- 기업·학교 네트워크 환경(방화벽)에서 안정적 동작
- 서버 자원 효율적 사용
- 확장성 (참여자 수 증가 대응)

## 고려한 옵션

### 옵션 A — P2P (Direct)
- 장점: 낮은 지연, 서버 부하 없음, 높은 프라이버시
- 단점: 참여자 N명 시 각자 N-1개 연결 유지 필요 (10명이면 각자 9개 연결), 기업·학교 방화벽 환경에서 차단 빈번, NAT Traversal 문제

### 옵션 B — MCU (Multipoint Control Unit)
- 장점: 클라이언트 부하 적음, 레이아웃 제어 용이
- 단점: 서버에서 모든 스트림을 믹싱하여 재전송 → 서버 자원 소모 큼, 지연 증가, 음질 손실

### 옵션 C — SFU (Selective Forwarding Unit) + MediaSoup (선택)
- 장점: 참여자 수 관계없이 서버와 1개 연결만 유지, 방화벽 환경 안정적(고정 IP·포트), 원본 품질 유지, MCU 대비 서버 부하 낮음
- 단점: 클라이언트 수신 대역폭 증가, 구현 복잡도 높음

## 결정
SFU 방식으로 WebRTC를 구현하고, 라이브러리는 MediaSoup을 사용한다.

MediaSoup 선택 이유는 다음과 같다.
- C++ 기반 코어 엔진으로 다수 동시 사용자 효율적 처리
- 최신 WebRTC 표준 완벽 지원, 주요 브라우저 호환
- Worker 프로세스 기반 아키텍처로 서버 리소스 효율적 활용
- 상용 서비스에서 검증된 라이브러리

## 근거
P2P 방식은 2명 통화에는 효율적이지만, 팀 단위 다중 사용자 환경에서는 연결 수가 기하급수적으로 증가한다. 10명이 참여하면 각자 9개 연결을 유지해야 하므로 클라이언트 부담이 매우 크다. SFU 방식에서는 참여자 수와 관계없이 각 사용자가 서버와 1개 연결만 유지하면 되어 확장성이 뛰어나다.

방화벽 문제도 중요한 고려사항이었다. 학교·기업 환경의 엄격한 방화벽 정책에서 P2P 연결은 차단되는 경우가 많지만, SFU는 서버의 고정 IP와 포트를 사용하므로 안정적인 연결이 가능하다.

## 트레이드오프
- 얻은 것: 다중 사용자 확장성, 방화벽 환경 안정성, 원본 음질 유지
- 감수한 것: 클라이언트 수신 대역폭 증가, SFU 서버 구현 복잡도

## 결과
- MediaSoup 기반 SFU 서버 구현 완료
- Transport, Producer, Consumer, Router, Room 구조로 설계
- OPUS 코덱 48kHz 고음질 오디오 지원
