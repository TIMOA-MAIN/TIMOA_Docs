# 0009. 채팅에 WebSocket 대신 REST + SSE 채택

## 상태
승인됨

## 맥락
채팅 기능 구현 시 실시간 메시지 수신을 위한 통신 방식을 결정해야 했다. 일반적으로 채팅 = WebSocket이라는 인식이 있지만, 채팅의 통신 패턴을 분석하면 반드시 WebSocket이 필요하지 않다는 판단이 있었다.

채팅의 통신 패턴은 다음과 같이 비대칭적이다.
- 서버 → 클라이언트: 새 메시지 도착 알림 (단방향, 빈번)
- 클라이언트 → 서버: 메시지 전송 (단방향, 간헐적)

Discord가 실제로 채팅에 REST + SSE 조합을 사용한다는 점을 참조했다.

## 고려한 옵션

### 옵션 A — WebSocket
- 장점: 양방향 실시간 통신, 낮은 지연
- 단점: 연결 유지 비용 높음, 채팅처럼 비대칭적 통신에 오버스펙, 서버 동시 연결 수 증가 시 부하 큼

### 옵션 B — REST + SSE (선택)
- 장점: 서버 → 클라이언트 단방향 푸시에 최적, HTTP 기반으로 방화벽·프록시 환경 안정적, 서버 부하 낮음, 자동 재연결 내장, 클라이언트 → 서버는 단순 REST POST
- 단점: 양방향이 아니라 클라이언트 → 서버 전송은 별도 REST 요청 필요, 초저지연이 필요한 경우 WebSocket 대비 불리

### 옵션 C — Polling
- 장점: 구현 단순
- 단점: 불필요한 서버 요청, 실시간성 낮음

## 결정
채팅 서버에 REST + SSE 조합을 채택한다.
- 메시지 전송: REST POST
- 메시지 수신: SSE (Server-Sent Events)

Discord의 아키텍처를 레퍼런스로 삼아 채팅의 통신 패턴이 본질적으로 비대칭임을 확인하고 이 방식을 선택했다.

## 근거
채팅에서 실시간성이 필요한 것은 서버 → 클라이언트 방향(새 메시지 알림)뿐이다. 클라이언트 → 서버 방향(메시지 전송)은 단순 HTTP POST로 충분하며, 이 경우 WebSocket의 지속 연결 유지 비용은 불필요한 오버헤드다. SSE는 HTTP 기반이므로 기존 인프라(Reverse Proxy, 방화벽)에서 별도 설정 없이 안정적으로 동작한다.

## 트레이드오프
- 얻은 것: 서버 부하 감소, HTTP 기반 안정성, 자동 재연결
- 감수한 것: 클라이언트 → 서버 전송이 REST 요청으로 분리됨

## 결과
- 채팅 서버 연결 유지 비용 감소
- HTTP 기반으로 Reverse Proxy에서 별도 WebSocket 처리 불필요
