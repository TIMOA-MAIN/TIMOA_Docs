# 0012. 대시보드 동기화에 REST + SSE 채택

## 상태
승인됨

## 맥락
대시보드(WBS, 칸반보드)는 여러 사용자가 동시에 사용하며, 한 사용자의 변경사항이 다른 사용자의 화면에 실시간으로 반영되어야 했다.

대시보드의 통신 패턴을 분석했다.
- 사용자가 섹션·Row·카드를 추가·수정·삭제하면 REST API로 DB에 저장
- 저장 완료 후 같은 채널의 다른 사용자들에게 변경 이벤트를 전파
- 클라이언트는 이벤트를 받아 UI를 즉시 업데이트

이 패턴은 서버 → 클라이언트 단방향 이벤트 전파가 핵심이다.

## 고려한 옵션

### 옵션 A — WebSocket
- 장점: 양방향 실시간 통신
- 단점: 대시보드의 통신 패턴이 단방향 이벤트 전파이므로 오버스펙, 연결 유지 비용

### 옵션 B — REST + SSE (선택)
- 장점: REST로 데이터 CRUD, SSE로 변경 이벤트 단방향 전파. 역할이 명확히 분리됨. HTTP 기반 안정성, 자동 재연결
- 단점: 클라이언트 → 서버 방향은 REST로 별도 처리

### 옵션 C — Polling
- 장점: 구현 단순
- 단점: 불필요한 서버 요청 발생, 실시간성 부족

## 결정
대시보드 서버에 REST + SSE 조합을 채택한다.

역할 분리는 다음과 같다.
- REST API: 섹션·Row·카드 CRUD
- SSE: 변경 이벤트 실시간 전파 (`CREATE_SECTION`, `UPDATE_ROW` 등)

## 근거
대시보드에서 실시간으로 필요한 것은 서버 → 클라이언트 방향의 변경 이벤트 전파뿐이다. 클라이언트 → 서버 방향은 사용자가 명시적으로 액션을 취할 때만 발생하므로 REST로 충분하다. 채팅([0009](./0009-chat-rest-sse.md))과 동일한 패턴으로 역할 분리가 명확하고 유지보수가 용이하다.

WebSocket을 사용하면 서버 부하 대비 얻는 이점이 없다. SSE는 HTTP 기반이므로 Reverse Proxy에서 별도 처리 없이 안정적으로 동작한다.

## 트레이드오프
- 얻은 것: 역할 분리 명확, 서버 부하 감소, HTTP 기반 안정성
- 감수한 것: 클라이언트 → 서버 전송이 REST로 분리됨

## 결과
- 대시보드 변경사항 실시간 동기화 구현 완료
- 섹션 삭제 시 하위 Row·Task 자동 삭제 (JPA Cascade) + SSE 이벤트 전파
- 관련 결정: [0004](../architecture/0004-protocol-mixed.md), [0009](./0009-chat-rest-sse.md)
