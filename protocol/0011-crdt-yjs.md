# 0011. 실시간 동시 편집에 CRDT(Y.js) 채택

## 상태
승인됨

## 맥락
여러 사용자가 동시에 같은 문서나 코드를 편집할 때 발생하는 충돌을 어떻게 해결할지 결정해야 했다. 이는 Google Docs, Notion 등 협업 툴의 핵심 기술적 과제다.

TIMOA에서 동시 편집이 필요한 영역은 다음과 같다.
- 문서 에디터 (Notionblock.js 기반)
- 코드 에디터 (Theia 기반)

두 사용자가 동시에 같은 위치를 편집할 때 최종 결과가 일관성 있게 병합되어야 하고, 커서 위치도 실시간으로 공유되어야 한다.

## 고려한 옵션

### 옵션 A — Lock 기반 (비관적 동시성 제어)
- 장점: 구현 단순, 충돌 자체를 방지
- 단점: 한 사용자가 편집 중이면 다른 사용자는 대기해야 함. 협업 UX 심각하게 저하

### 옵션 B — OT (Operational Transformation) 직접 구현
- 장점: Google Docs 등에서 검증된 방식
- 단점: 알고리즘 구현 복잡도 매우 높음, 엣지 케이스 처리 어려움, 검증에 많은 시간 필요

### 옵션 C — CRDT + Y.js (선택)
- 장점: 분산 환경에서 수학적으로 충돌 없는 병합 보장, 검증된 라이브러리, React·Node.js 생태계 호환성 높음, 네트워크 단절 후 자동 상태 복원
- 단점: CRDT 개념 학습 비용, 메모리 사용량 OT 대비 다소 높음

## 결정
CRDT 방식을 채택하고 구현체로 Y.js를 사용한다.

Y.js 핵심 구성은 다음과 같다.
- `Y.Doc`: CRDT 문서 상태 관리 객체
- `Y.Text`: 텍스트 편집용 CRDT 타입 (Operation-based CRDT)
- `Y.Map`: 키-값 저장용 CRDT 타입
- `y-websocket`: WebSocket 기반 실시간 동기화
- `y-monaco`: Monaco Editor 연동

## 근거
CRDT(Conflict-free Replicated Data Type)는 분산 환경에서 수학적으로 충돌 없는 병합을 보장하는 데이터 구조다. OT를 직접 구현하는 것보다 검증된 Y.js를 사용하는 것이 안정성, 개발 속도, 유지보수 모든 면에서 유리했다.

Y.js는 Notion, Linear 등 실제 프로덕션 서비스에서 사용하는 검증된 라이브러리이며, React와 Node.js 생태계에 최적화되어 있어 TIMOA 기술 스택과 잘 맞는다.

## 트레이드오프
- 얻은 것: 충돌 없는 동시 편집, 커서 위치 공유, 네트워크 단절 후 자동 복원
- 감수한 것: CRDT 학습 비용, OT 대비 다소 높은 메모리 사용

## 결과
- 실시간 동시 편집 기능 구현 완료
- 여러 사용자의 커서 위치 실시간 표시
- DB 저장 및 재연결 시 문서 상태 자동 복원
