# 0004. 서비스별 통신 프로토콜 혼용 결정 (표준화 포기)

## 상태
승인됨

## 맥락
MSA 구조에서 서비스 간, 클라이언트-서버 간 통신 방식을 어떻게 통일할지 결정이 필요했다. SOA처럼 단일 표준 프로토콜을 강제할 것인지, 각 서비스 특성에 맞는 프로토콜을 자유롭게 선택할 것인지가 핵심 질문이었다.

각 서비스의 통신 특성이 근본적으로 달랐다.
- 실시간 동시 편집: 지속 연결, 양방향, 저지연 필수
- 음성 통화: 미디어 스트리밍, UDP 기반
- 채팅: 단방향 서버 푸시, 간헐적 클라이언트 전송
- 대시보드: 상태 변경 알림, 단방향 서버 푸시
- 코드 실행: 요청-응답, 비동기 출력 스트리밍

## 고려한 옵션

### 옵션 A — 단일 프로토콜 표준화 (REST 또는 WebSocket)
- 장점: 일관성, 유지보수 단순, 학습 비용 낮음
- 단점: 모든 서비스 특성을 하나의 프로토콜로 처리하면 비효율 발생. 음성 통화를 REST로 처리하거나, 단순 조회를 WebSocket으로 처리하는 것은 과도한 오버헤드

### 옵션 B — 서비스별 최적 프로토콜 선택 (선택)
- 장점: 각 서비스 특성에 최적화, 성능과 구현 효율 극대화
- 단점: 프로토콜 다양성으로 전체 시스템 파악 어려움, 신규 팀원 학습 비용 증가

## 결정
서비스별 특성에 따라 최적 프로토콜을 독립적으로 선택한다.

| 서비스 | 프로토콜 | 선택 이유 |
|--------|----------|-----------|
| CRDT Server | WebSocket (Y.js) | 지속 연결, 양방향 실시간 동기화 |
| Voice Server | WebRTC + MediaSoup | 미디어 스트리밍, UDP 기반 저지연 |
| Chat Server | REST + SSE | 단방향 푸시, HTTP 기반 안정성 |
| Dashboard Server | REST + SSE | 상태 변경 알림, 단방향 푸시 |
| Compiler Server | REST | 요청-응답, 비동기 출력 |
| LSP Server | WebSocket + JSON-RPC | LSP 표준 프로토콜 준수 |

## 근거
서비스마다 통신 패턴이 본질적으로 달랐다. 실시간 동시 편집은 양방향 지속 연결이 필수이고, 채팅은 서버 푸시만 있으면 충분하며, 음성은 미디어 스트리밍 특성상 WebRTC가 유일한 현실적 선택이다. 단일 프로토콜로 강제하면 일부 서비스에서 불필요한 복잡성이 생긴다.

## 트레이드오프
프로토콜 다양성으로 인해 전체 시스템을 이해하는 데 더 많은 맥락이 필요하다. 각 프로토콜 선택의 근거는 개별 MADR([0009](../protocol/0009-chat-rest-sse.md), [0010](../protocol/0010-voice-sfu-mediasoup.md), [0011](../protocol/0011-crdt-yjs.md), [0012](../protocol/0012-dashboard-rest-sse.md))에 상세히 기록한다.

## 결과
- 각 서비스가 통신 특성에 최적화된 프로토콜을 사용
- 전체 시스템에서 REST, WebSocket, WebRTC, SSE, JSON-RPC가 혼용됨
- 관련 결정: [0009](../protocol/0009-chat-rest-sse.md), [0010](../protocol/0010-voice-sfu-mediasoup.md), [0011](../protocol/0011-crdt-yjs.md), [0012](../protocol/0012-dashboard-rest-sse.md)
