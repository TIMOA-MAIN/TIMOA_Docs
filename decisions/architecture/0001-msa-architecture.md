# 0001. MSA 지향 아키텍처 채택

## 상태
승인됨

## 맥락
TIMOA는 실시간 코딩 협업 플랫폼으로, 음성 통화, 실시간 채팅, 코드 편집, 문서 협업, 대시보드 등 기능적으로 독립적인 여러 도메인을 하나의 플랫폼에서 제공해야 했다.

초기 설계 단계에서 다음과 같은 문제를 고려했다.

- 각 기능 도메인이 요구하는 기술 스택이 상이함 (실시간 음성 처리 vs 문서 동기화 vs 코드 컴파일)
- 특정 기능의 트래픽 급증이 전체 시스템에 영향을 주지 않아야 함
- 팀원별 독립적인 개발과 배포가 필요함
- Visual Studio Live Share, Discord, Notion, Trello의 기능을 통합하는 복잡한 도메인 구조

모놀리식 아키텍처, SOA, MSA 중 어떤 아키텍처를 선택할지 결정이 필요했다.

## 고려한 옵션

### 옵션 A — 모놀리식 아키텍처
- 장점: 초기 개발 속도 빠름, 단일 배포, 팀 러닝커브 낮음
- 단점: 기능별 기술 스택 분리 불가, 특정 기능 장애 시 전체 영향, 확장 어려움

### 옵션 B — SOA
- 장점: 서비스 재사용성, 표준화된 인터페이스
- 단점: ESB 도입 필요, 실시간 처리(WebSocket, WebRTC)에 부적합한 SOAP 기반 구조, 운영 복잡도 높음

### 옵션 C — MSA (선택)
- 장점: 서비스별 독립 기술 스택, 독립 배포, 장애 격리, 기능별 독립 확장
- 단점: 분산 시스템 복잡성, 서비스 간 통신 설계 필요, 운영 인프라 요구

## 결정
MSA를 지향하는 아키텍처를 채택한다. 단, 예산과 온프레미스 인프라 제약으로 인해 완전한 MSA가 아닌 MSA 지향 구조로 구현한다.

서비스 분리 단위는 다음과 같다.
- CRDT Server (실시간 동시 편집)
- Voice Server (음성 통화)
- Compiler Server (코드 실행)
- LSP Server (인텔리센스)
- Dashboard Server (대시보드·칸반)
- Chat Server (채팅)
- Proxy Server (Reverse Proxy · LB · 헬스체크)

## 근거
각 서비스가 요구하는 기술이 근본적으로 달랐다. 음성 처리는 Node.js + MediaSoup, 문서 동시 편집은 Y.js CRDT, 코드 컴파일은 Java Spring Boot + GCC가 각각 최적이었다. 이를 하나의 모놀리식으로 구성하면 기술 선택의 자유도가 지나치게 제한된다.

SOA를 선택하지 않은 이유는 WebSocket, WebRTC 같은 실시간 연결이 핵심 기능인 시스템에서 SOAP 기반의 ESB는 구조적으로 부적합하기 때문이다.

## 트레이드오프
현재 단일 MS SQL DB를 공유하고 있어 완전한 MSA 원칙에 미달한다. DB 분리, Redis 서비스 간 통신, 컨테이너화가 추가되면 완전한 MSA로 전환 가능하다.

## 결과
- 서비스별 독립적인 기술 스택 선택 가능
- 팀원별 담당 서비스 독립 개발 가능
- 공유 DB로 인한 결합이 남아있어 추후 DB 분리 필요
