# 0003. 단일 공유 DB(MS SQL) 유지 결정

## 상태
미완성

## 맥락
MSA 원칙에 따르면 각 서비스는 독립적인 DB를 가져야 한다. 그러나 TIMOA 프로젝트는 예산과 온프레미스 서버 가용 환경의 제약으로 인해 완전한 DB 분리를 구현하지 못했다.

원래 목표했던 설계는 다음과 같았다.
- 서비스별 독립 DB 인스턴스
- Redis를 통한 서비스 간 이벤트 통신
- AWS RDS 또는 각 서비스에 최적화된 DBMS 선택 (채팅 → MongoDB, 세션 → Redis 등)

## 고려한 옵션

### 옵션 A — 서비스별 독립 DB (목표)
- 장점: 완전한 MSA 원칙 준수, 서비스별 최적 DBMS 선택 가능, 장애 격리
- 단점: 인프라 비용 증가, 분산 트랜잭션 처리 복잡성, 운영 부담

### 옵션 B — 단일 공유 MS SQL (선택)
- 장점: 인프라 비용 최소화, ACID 트랜잭션 보장, 관리 단순함
- 단점: MSA 원칙 미준수, 공유 DB가 단일장애점, 서비스 간 결합 발생, 확장 한계

## 결정
예산과 인프라 제약으로 단일 MS SQL DB를 모든 서비스가 공유하는 구조를 유지한다. 이는 임시 결정이며 추후 인프라 환경이 개선되면 DB 분리를 진행한다.

## 근거
온프레미스 서버 가용 환경의 한계로 다수의 DB 인스턴스를 운용하기 어려웠다. 또한 하드웨어 LB, Redis 클러스터 등 추가 인프라도 같은 이유로 도입하지 못했다. 기능 구현을 우선시하는 현실적 판단이었다.

## 트레이드오프
- 현재: ACID 보장, 관리 단순, 인프라 비용 최소화
- 감수: MSA 원칙 미준수, 공유 DB 병목 가능성, 서비스 간 스키마 결합

## 개선 방향
인프라 환경 개선 시 다음 순서로 DB 분리를 진행한다.
1. Chat Server → MongoDB (읽기/쓰기 빈번한 메시지 저장)
2. CRDT Server → Redis (휘발성 세션 데이터)
3. Dashboard Server → 독립 MS SQL 인스턴스
4. 나머지 서비스 순차 분리

## 결과
- 단기: 안정적인 ACID 트랜잭션과 낮은 운영 비용
- 장기: DB 분리 미완성으로 완전한 MSA 미달
