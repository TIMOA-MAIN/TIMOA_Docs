# 0007. 문서 에디터 단순 React 컴포넌트 → Notionblock.js 전환

## 상태
승인됨

## 맥락
초기 문서 에디터는 제목, 부제목, 텍스트, 이미지 블록을 React 컴포넌트로 직접 구현했다. 기본적인 블록 추가·삭제·순서 변경은 가능했지만 UX 측면에서 한계가 있었다.

사용자 경험 개선 요구사항은 다음과 같았다.
- Notion과 유사한 블록 기반 에디터 UX
- 슬래시 커맨드(`/`)로 블록 타입 선택
- 드래그 앤 드롭 블록 재정렬
- 블록 타입별 스타일 일관성
- 마크다운 단축키 지원

이를 직접 구현하면 에디터 UX 개발에만 상당한 공수가 필요했다.

## 고려한 옵션

### 옵션 A — React 컴포넌트 직접 구현 유지 및 기능 확장
- 장점: 완전한 제어권, 외부 의존성 없음
- 단점: 슬래시 커맨드, 드래그 앤 드롭 등 UX 기능 직접 구현 공수 높음, 유지보수 부담

### 옵션 B — Notion 외부 API 연동
- 장점: Notion의 완성된 에디터 UX 그대로 활용
- 단점: Notion API는 관리자 API Key가 필요하고 외부 서비스 의존성 발생. 실제 시도했으나 관리자 API Key 제약으로 실패 → [0015](../integration/0015-notion-api-fallback.md) 참조

### 옵션 C — Notionblock.js 도입 (선택)
- 장점: Notion과 유사한 블록 에디터 UX 제공, 슬래시 커맨드·드래그 앤 드롭 내장, 빠른 적용 가능
- 단점: 외부 라이브러리 의존성, 세부 커스터마이징 범위 제한

## 결정
Notionblock.js를 도입하여 문서 에디터를 전환한다.

## 근거
Notion API 연동 실패([0015](../integration/0015-notion-api-fallback.md)) 이후 Notion과 유사한 UX를 내부적으로 구현하는 방향으로 전환했다. Notionblock.js는 Notion 스타일의 블록 에디터를 빠르게 적용할 수 있는 라이브러리로, 직접 구현 대비 개발 공수를 크게 줄이면서 목표한 UX를 달성할 수 있었다.

## 트레이드오프
- 얻은 것: Notion 유사 UX, 슬래시 커맨드, 드래그 앤 드롭, 개발 공수 절감
- 감수한 것: 외부 라이브러리 의존성, 세부 커스터마이징 한계

## 결과
- 블록 기반 문서 에디터 UX 완성
- 사용자 친화적인 Notion 유사 인터페이스 제공
- 관련 결정: [0005](./0005-document-server-migration.md), [0015](../integration/0015-notion-api-fallback.md)
