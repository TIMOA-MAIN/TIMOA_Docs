# 0015. Notion 외부 API 연동 시도 → 관리자 API Key 제약으로 내부 구현 전환

## 상태
대체됨

## 맥락
문서 채널 기능을 구현할 때 Notion과 유사한 UX를 제공하는 것이 목표였다. 직접 에디터를 구현하는 대신 Notion의 완성된 에디터를 외부 API로 연동하는 방안을 먼저 검토했다.

Notion API를 통해 다음 기능을 연동하려 했다.
- 페이지 생성·수정·삭제
- 블록 단위 콘텐츠 관리
- 팀 워크스페이스 공유

## 시도한 내용

Notion API는 공개 API를 제공하지만, 실제 연동 시도 과정에서 다음 제약이 발견됐다.

- Notion API 연동은 워크스페이스 관리자 권한의 API Key가 필요
- 외부 서비스에서 Notion 워크스페이스에 접근하려면 각 사용자가 개별 Notion 계정과 API Key를 발급·관리해야 함
- 플랫폼 내에서 통합된 문서 경험을 제공하려면 모든 사용자가 Notion 계정을 보유해야 하는 외부 의존성 발생
- 관리자 API Key를 서버에서 공유하는 방식은 Notion 정책상 허용되지 않음

결론적으로 Notion API로는 TIMOA 내에서 통합된 문서 경험을 제공하는 것이 불가능했다.

## 결정
Notion 외부 API 연동을 포기하고 내부 구현으로 방향을 전환한다.

대안으로 Notionblock.js를 도입하여 Notion과 유사한 블록 기반 에디터 UX를 내부적으로 구현한다. ([0007](../service/0007-notionblock-adoption.md) 참조)

## 근거
외부 서비스 API 연동은 항상 해당 서비스의 정책과 제약에 종속된다. Notion의 API Key 정책은 플랫폼 수준의 통합보다 개인 자동화 용도에 최적화되어 있어 TIMOA의 요구사항과 맞지 않았다. 내부 구현으로 전환함으로써 외부 의존성을 제거하고, 사용자가 별도 계정 없이 바로 문서 기능을 사용할 수 있게 되었다.

## 교훈
외부 API 연동 전 해당 서비스의 인증 정책과 사용 제약을 먼저 검토해야 한다. 특히 관리자 권한이 필요한 API는 B2C 플랫폼에서 활용하기 어렵다.

## 트레이드오프
- 얻은 것: 외부 서비스 의존성 제거, 사용자 별도 계정 불필요, 완전한 제어권
- 감수한 것: Notion 완성된 에디터 대신 내부 구현 필요, 추가 개발 공수

## 결과
- Notion API 연동 시도 실패 및 포기
- Notionblock.js 기반 내부 구현으로 대체 ([0007](../service/0007-notionblock-adoption.md))
- 사용자 경험 목표는 달성, 외부 의존성 없음
- 관련 결정: [0007](../service/0007-notionblock-adoption.md)
